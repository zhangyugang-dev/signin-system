# CR-001：签到模式变更 — 支持公开身份签到

## 1. 需求变更说明

### 变更背景
当前签到逻辑要求用户必须在管理员预导入的参会名单中才能签到。实际需求是：**签到页面支持公开身份签到，不要求在名单中**。任何人输入姓名+手机号后四位即可完成签到。

### 变更内容
| 项目 | 变更前 | 变更后 |
|------|--------|--------|
| 签到前置条件 | 必须在名单中（name + phoneLast4 匹配） | 无前置条件，任何人可签到 |
| 签到逻辑 | 查名单 → 匹配 → 签到 | 直接创建签到记录（去重：同 name+phoneLast4 不重复签到） |
| 数据来源 | 仅管理员 CSV 导入 | CSV 导入 + 现场自助签到均产生记录 |
| Attendee 模型 | phone 必填（完整手机号） | phone 改为可选（公开签到只有后四位） |
| 管理后台统计 | 已签到/未签到（基于名单总数） | 总签到人数（含名单+现场） |

### 不变更的部分
- 管理员仍可导入 CSV 名单（含完整手机号）
- 已签到不可重复签到
- CSV 导出功能保持不变
- 活动配置功能不变

## 2. 技术调整方案

### 2.1 数据库层（Prisma Schema）
- `Attendee.phone`：改为可选字段（`String?`），因为公开签到用户不会提供完整手机号
- 新增字段 `Attendee.source`：标识记录来源（`"import"` 或 `"walkin"`）

### 2.2 后端逻辑（actions.ts）
- `checkIn(name, phoneLast4)`：
  - 变更前：查找名单匹配 → 不在名单则拒绝
  - 变更后：
    1. 先查是否已有该 name+phoneLast4 的已签到记录 → 有则提示已签到
    2. 查是否有预导入的未签到记录匹配 → 有则更新为已签到
    3. 都没有 → 创建新的 walkin 签到记录，直接标记已签到

### 2.3 前端页面
- 签到页面（page.tsx）：无需改动表单，只需调整提示文案
- 管理后台（admin/page.tsx）：
  - 统计卡片调整：增加"现场签到"计数
  - 参会人列表增加"来源"列

### 2.4 CSV 导出
- 增加"来源"列（导入/现场）

## 3. 测试用例

### TC-01：公开签到 — 非名单用户签到成功
- **前置**：名单中无"张三 / 1234"
- **操作**：签到页输入姓名"张三"、后四位"1234"，点签到
- **预期**：提示"签到成功"，管理后台出现该记录，来源为"现场"

### TC-02：公开签到 — 重复签到拦截
- **前置**：TC-01 已完成
- **操作**：再次输入"张三"+"1234"签到
- **预期**：提示"已签到，不可重复签到"

### TC-03：名单用户签到
- **前置**：管理员导入 CSV，名单中有"李四 / 13900001234"
- **操作**：签到页输入"李四"+"1234"
- **预期**：签到成功，管理后台该记录状态变为已签到，来源为"导入"

### TC-04：名单用户重复签到拦截
- **前置**：TC-03 已完成
- **操作**：再次输入"李四"+"1234"
- **预期**：提示"已签到，不可重复签到"

### TC-05：统计数据正确性
- **前置**：TC-01 和 TC-03 均完成
- **操作**：查看管理后台统计
- **预期**：总人数 ≥ 2，已签到 ≥ 2，现场签到 ≥ 1

### TC-06：CSV 导出包含来源
- **操作**：管理后台点击"导出 CSV"
- **预期**：CSV 包含"来源"列，值为"导入"或"现场"

### TC-07：空姓名/不足四位拦截
- **操作**：签到页姓名留空或后四位不足4位，点签到
- **预期**：前端拦截，不发请求
